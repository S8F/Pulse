# Pulse — Frontend Developer Take-Home Assessment

## Overview

Welcome, and thanks for taking the time to work through this assessment.

**Pulse** is a small internal team status dashboard — a lightweight tool where team members post daily status updates and managers can view and filter them. We've generated the initial React code for this app using Figma Make. Your job is to take that generated code, refactor it into clean, production-quality components, and wire it up to a mock API.

The root app in this repo is intentionally only a scaffold right now: it boots Vite, TanStack Router, TanStack Query, Tailwind, and MSW, and the `/` route renders a placeholder screen. The generated Figma export lives under `figma-output/` at the repo root as reference material for the screens you'll build into the app.

This assessment is designed to closely mirror the actual day-to-day work of this role. We're not looking for tricks, gotchas, or algorithmic puzzles. We want to see how you work with real-world code.

**Time expectation:** 4–6 hours. Quality over completeness — we'd rather see two screens done well than three done hastily. Here's the bottom line: there is a lot to do in this task, and I do not expect most candidates to finish everything. What I'm looking for is an example of your very best work. If you complete everything but the code is garbage, you will *not receive a callback. However, if you only complete two screens and the code is exceptional, there is a high probability of moving forward. If you're running over 6 hours, stop and submit what you have with a note about what you would have done next.

## The Stack

This repo comes pre-configured with the tools and libraries we use in production:

- **React 18** with TypeScript
- **TanStack Router** for file-based routing
- **TanStack Query** for server state management
- **Tailwind CSS v4** for styling (CSS-based config, no `tailwind.config.ts`)
- **Vite** for dev server and bundling
- **Biome** for linting and formatting (our config is included — please use it)
- **MSW** for mock API (runs in the browser via service worker)
- **pnpm** as the package manager

Also included and available for your use:

- **@base-ui/react** — headless UI primitives
- **lucide-react** — icon library
- **tailwind-variants** — type-safe component variants
- **clsx** + **tailwind-merge** — class name utilities (see `src/lib/cn.ts`)

You are welcome to add small utility libraries if you feel they're warranted (e.g., a date formatting lib, a form library), but please don't introduce a new state management framework or CSS approach. Part of the assessment is working within constraints.

One repo-specific note: `figma-output/` includes the raw Figma export and its own standalone Vite project files (`package.json`, `package-lock.json`, and a local `node_modules/`). Treat that folder as reference input, not as the app you should run or submit.

Run those commands from the repo root. The `figma-output/README.md` mentions `npm`, but that applies only to the standalone Figma export inside that folder, not to this assessment app.

## The Task

### Part 1 — Refactor the Figma Output (~1.5 hours)

The files in `figma-output/` are raw Figma Make output. They work visually, but they're not production code. Your job:

1. **Decompose** them into clean, reusable components. Extract things that should obviously be shared — status chips, card layouts, metric cards, form fields, the nav sidebar, etc.
2. **Organize** them into a sensible file/folder structure of your choosing.
3. **Clean up** the styling. The Figma output likely uses inline styles or generated class names. Convert to Tailwind utility classes. Match the reference designs in `designs/`, but pixel-perfection is not required — we care about responsive, maintainable CSS.
4. **Add TypeScript types** where the generated code is loosely typed or uses `any`.

We're evaluating: component decomposition, naming, prop design, Tailwind fluency, and TypeScript usage.

### Part 2 — Wire Up the API (~2 hours)

The mock API spec is in `api-spec.md`. The MSW handlers in `src/api/mock-server.ts` implement it. Your job:

1. **Integrate TanStack Query** to fetch and mutate data. The status feed should use the paginated list endpoint. The form should POST/PUT to create or update a status.
2. **Handle loading and error states** — show appropriate UI for both. Don't just ignore the unhappy path.
3. **Implement cache invalidation or optimistic updates** when a status is created or updated. We want to see that you understand how TanStack Query's cache works.
4. **Wire up the filters** on the feed screen (status category and team) as query parameters.
5. **Connect the routes** — the root app currently only defines `/`, so the feed, form (new and edit modes), and team overview should become navigable TanStack Router file routes under `src/routes/`.

We're evaluating: TanStack Query patterns, error/loading handling, cache strategy, and routing integration.

### Part 3 — Polish & Decisions (~1 hour)

1. **Form validation**: Add client-side validation to the status form. Project is required. Update body has a minimum length. Show inline error messages.
2. **Empty state**: The team overview screen should handle the case where no updates exist. Use the empty state from the design.
3. **Responsive behavior**: The feed should work reasonably on a tablet-width viewport (~768px). The sidebar can collapse or become a top nav — your call.
4. **DECISIONS.md**: Write a short document (a few paragraphs, not an essay) explaining 2–3 significant decisions you made and why. This could be about component structure, state management approach, a tradeoff you navigated, or something you'd do differently with more time.

We're evaluating: attention to detail, judgment, communication skills, and self-awareness about tradeoffs.

## API Quick Reference

Full details are in `api-spec.md`, but here's the shape of things:

| Endpoint | Method | Description |
|---|---|---|
| `/api/statuses` | GET | Paginated list. Supports `?page`, `?status`, `?team`, `?search` |
| `/api/statuses/:id` | GET | Single status by ID |
| `/api/statuses` | POST | Create a new status |
| `/api/statuses/:id` | PUT | Update an existing status |
| `/api/statuses/:id` | DELETE | Delete a status |
| `/api/teams` | GET | List of all teams |
| `/api/team-summary` | GET | Aggregated metrics for the team overview |

## FAQ

**Can I use a component library like shadcn/ui or Radix?**
For primitives like dropdowns, date pickers, and modals — yes, that's fine and even encouraged. For layout and styling — use Tailwind directly.

**Can I change the API spec or mock server?**
No. Treat it as a real backend you can't control. If you find something awkward about the API shape, mention it in your DECISIONS.md — that's valuable signal to us.

**What if I can't finish everything?**
That's okay. Prioritize Parts 1 and 2. In your DECISIONS.md, note what you'd tackle next. We're much more interested in the quality of what you do complete.

**Can I use AI tools (Copilot, ChatGPT, etc.)?**
Yes. We assume you will. This is representative of how we actually work. However — you should be able to explain every line of code you submit. If we ask "why did you structure it this way?" during the review conversation, "the AI suggested it" is not a sufficient answer.
